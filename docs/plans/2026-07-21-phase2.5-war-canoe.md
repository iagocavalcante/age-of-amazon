# Phase 2.5 — War Canoe & Water Movement Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add a **water-borne movement domain** and the **War Canoe** — a ranged
water combat unit launched from a shore **Dock** — completing what Phase 2
deferred. Land units are unchanged; water units travel on water (deep +
shallow) and cannot enter land.

**Architecture:** Movement gains a second **domain** axis alongside the
owner-awareness axis added by the palisade gate. The pathfinder already threads
`player_id` (default `NO_OWNER`); this adds a `water: bool` (default `false` =
land) threaded the same way. `WorldData` gains a `NAVIGABLE` table (water
biomes) consulted when `water` is true, mirroring how `WALKABLE` is consulted
for land. A unit's domain is **data-derived** from its def (`"water": true`),
so it's deterministic and replicates/saves for free. The **Dock** is a normal
building with a new `requires_adjacent_water` placement constraint; its
`_spawn_unit` launches water units onto an adjacent navigable cell. The War
Canoe is **ranged** (combat is distance-based), so it fights shore units and
other canoes without pathing onto land. The AI stays land-focused (it ignores
docks/canoes — documented, not broken).

**Tech Stack:** Godot 4.5, GDScript. Verification via headless `--test-*`
harnesses (ADR 12) + `tools/test_mp.sh`; movie-writer capture for visuals.

**Grounding facts (already verified):**
- `Constants` (from Phase 2): `WALKABLE`/`BUILDABLE`/`MOVEMENT_COST` keyed by biome. Water: `WATER_SHALLOW` walkable (cost 2.0), `WATER_DEEP` not walkable (INF). `NO_OWNER = -1` in WorldData.

- `Pathfinder`: `find_path_world(from, to, player_id=NO_OWNER)` → `_nearest_walkable(cell, player_id)` → `_astar(start, goal, player_id)` → `_neighbors(cell, player_id)`; `adjacent_walkable(footprint, near)` uses `is_walkable`.

- `Building._spawn_unit(unit_type)` finds an `adjacent_walkable` cell near the footprint and spawns there. `_exec_place` validates buildable/occupied/resource/explored/era.

func _run_water_test() -> void:

	# Find a deep-water cell and a land cell near origin (scan).
	var water_cell := Vector2i(999999, 999999)

**Step 2: Run → FAIL** (`is_walkable_for` has no water param yet).

**Step 3: Implement**

`Constants.gd` — a NAVIGABLE table (water domain), built in `_ready` beside WALKABLE:
```gdscript
# Which biomes a WATER unit can travel (the inverse domain of WALKABLE). Only
# open water; land and unwalkable water-edge cliffs are not navigable.
NAVIGABLE = {
	Biome.WATER_DEEP: true,
	Biome.WATER_SHALLOW: true,
}
```
Declare `var NAVIGABLE: Dictionary = {}` with the other tables. (Non-water biomes absent → `.get(biome, false)` = false.) Also a movement cost for water travel — reuse MOVEMENT_COST (shallow 2.0) or add a flat water cost; simplest: water units use a flat cost (e.g. 1.0) on navigable cells. Define `const WATER_MOVE_COST := 1.0` (canoes glide) and use it in movement_cost_for for the water domain.

`WorldData.gd` — extend the `_for` functions with a `water` param (default false = land, so all existing callers unchanged):
```gdscript
func is_walkable_for(cell: Vector2i, player_id: int, water: bool = false) -> bool:
	if water:
		if occupied.has(cell):
			return false   # buildings block water too (no water buildings yet)
		return Constants.NAVIGABLE.get(get_biome(cell), false)
	# --- land domain (unchanged) ---
	if occupied.has(cell):
		if player_id != NO_OWNER and _is_own_gate(occupied[cell], player_id):
			return true
		return false
	return Constants.WALKABLE.get(get_biome(cell), false)

func movement_cost_for(cell: Vector2i, player_id: int, water: bool = false) -> float:
	if water:
		if occupied.has(cell):
			return INF
		return Constants.WATER_MOVE_COST if Constants.NAVIGABLE.get(get_biome(cell), false) else INF
	# --- land domain (unchanged) ---
	if occupied.has(cell):
		if player_id != NO_OWNER and _is_own_gate(occupied[cell], player_id):
			return Constants.MOVEMENT_COST.get(get_biome(cell), INF)
		return INF
	return Constants.MOVEMENT_COST.get(get_biome(cell), INF)
```
Keep base `is_walkable`/`movement_cost` delegating with `(cell, NO_OWNER)` → land (the `water` default false).

`Pathfinder.gd` — thread `water` (default false) through the chain: `find_path_world(from, to, player_id=NO_OWNER, water=false)` → `_nearest_walkable(cell, player_id, water)` → `_astar(start, goal, player_id, water)` (uses `movement_cost_for(neighbor, player_id, water)`) → `_neighbors(cell, player_id, water)` (all `is_walkable_for(±, player_id, water)`). Also add `adjacent_walkable(footprint, near, water=false)` — its internal `is_walkable(candidate)` becomes `is_walkable_for(candidate, NO_OWNER, water)` (so a water=true call finds an adjacent NAVIGABLE cell — used by the Dock in Task 2).

`Unit.gd` — a unit's domain, data-derived:
```gdscript
var is_water_unit: bool = false   # set in _ready from the def
```
In `_ready`, `is_water_unit = Constants.UNIT_DEFS.get(unit_type, {}).get("water", false)`. In `_start_path_to`, pass it: `find_path_world(global_position, target, player_id, is_water_unit)`. (Confirm all movement funnels through `_start_path_to` — it does per B3b.)

**Step 4: Run → PASS** (all four `[test-water]` lines). Then `--test-move`, `--test-systems`, `--test-palisade-gate` → still OK (default `water=false` keeps land pathing byte-identical; gate still works).

**Step 5: Commit** `feat(water): water movement domain — NAVIGABLE + water-threaded pathfinding`.

## Task 2: Dock building (shore placement + domain-aware unit launch)

**Files:** `scripts/autoloads/Constants.gd`, `scripts/autoloads/CommandRouter.gd`, `scripts/autoloads/GameManager.gd` (`find_buildable_cell`), `scripts/buildings/Building.gd` (`_spawn_unit`), `scripts/art/BuildingArtist.gd`, `scripts/autoloads/AssetLibrary.gd`, `scripts/ui/HUD.gd`, `scripts/main/Main.gd`.

**Def:** `BUILDING_DEFS["dock"]`: footprint (2,2), cost {WOOD 80}, era 2, max_hp 350, `trains: ["war_canoe"]`, vision_tiles 6, and a new key `"requires_adjacent_water": true`.

**Shore placement constraint:** in `CommandRouter._exec_place`, after the era gate and buildable/occupied checks, if `def.get("requires_adjacent_water", false)`, require at least one cell orthogonally adjacent to the footprint to be a water biome (WATER_SHALLOW or WATER_DEEP). Reject otherwise. Mirror the same check in `GameManager.find_buildable_cell` (so the AI/harness only get valid dock cells). Add a small `WorldData` helper `is_water(cell) -> bool` (biome is WATER_SHALLOW/WATER_DEEP) to share the check.

**Domain-aware launch:** in `Building._spawn_unit`, derive the spawned unit's domain from its def and find the right adjacent cell:
```gdscript
	var water_unit: bool = Constants.UNIT_DEFS.get(unit_type, {}).get("water", false)
	var spot: Dictionary = GameManager.pathfinder.adjacent_walkable(footprint_cells, <near>, water_unit)
```
So a water unit (canoe) spawns on an adjacent NAVIGABLE cell (the dock's water side); land units unchanged.

**Art + HUD:** `BuildingArtist.build_dock(color)` (a pier/jetty over water), register in AssetLibrary; add "dock" to the HUD build menu + a FREE `BUILD_KEYS` entry (used: B/N/T/S/P/K/M — pick e.g. KEY_D for Dock; verify free). Era-gated (era 2) automatically.

Assert: (a) `requires_adjacent_water` rejects a dock placed inland (no adjacent water) — build count unchanged; (b) a dock placed on the shore succeeds; (c) the dock trains a war_canoe (queue_train), and after it completes the canoe SPAWNS on a navigable (water) cell (assert the spawned unit's cell `is_water`). Read `_run_build_test`/B2's era-advance for the Chiefdom setup and how to drive train completion (or call `_spawn_unit` semantics). Deterministic.
